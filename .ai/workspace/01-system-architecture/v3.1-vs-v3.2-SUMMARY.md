# Sovereign Workspace v3.1 → v3.2: Key Changes Summary
**Document:** Quick reference for product + engineering alignment  
**Length:** ~5 min read  
**Format:** Problem → Solution → Impact

---

## EXECUTIVE SUMMARY

**v3.1 Issue:** Agent responsibilities overlapped. When `/sync` failed, it was unclear whether `scraper-agent` or `delta-sync-agent` should retry. Data ownership was ambiguous (who writes `sync-status.json`?). This caused unpredictable failures, token waste on reprocessing, and scaling friction.

**v3.2 Solution:** Established **Single Responsibility Principle** + **Explicit Data Ownership** + **Clear Error Recovery Cascades**.

**User Impact:** Zero. All commands work identically; infrastructure is clearer.  
**Engineering Impact:** Easier to debug, test, scale, and add new features.

---

## PROBLEM BREAKDOWN

| v3.1 Gap | Why It Matters | Consequence | v3.2 Fix |
|---------|----------------|-------------|----------|
| **Sub-agent overlap:** `delta-sync-agent` + `scraper-agent` both claim URL hashing | Unclear who owns delta detection | Duplicate work, missed edge cases, silent failures | Single `delta-detector` sub-agent owns this; `scraper-agent` is primary owner |
| **Data file ambiguity:** `sync-status.json` written by both scrapers | State corruption, stale timestamps | Failed syncs, re-scraping old data | **Only** `scraper-agent` writes; others read-only |
| **Error recovery unclear:** 429 on scrape — retry in `scraper-agent` or `ethics-compliance`? | Inconsistent retry logic | Some failures auto-retry, others don't; unpredictable | Clear cascade: `ethical-crawler` sub-agent retries with backoff; if 3x fails, `scraper-agent` logs & skips |
| **Memory layer undefined:** Context bloat, unclear who compresses | Token waste, slow context loading | Session state lost, reprocessing old files | `memory-manager` utility agent owns all compression; guides agent calls it before/after commands |
| **Content flow ambiguity:** Voice rules applied by `creator-agent` AND validated by `brand-agent` | Tone drift, looping reprocessing | Content approved, then rejected in `/review` for tone | `creator-agent` creates; `brand-voice-applier` sub-agent validates immediately; blocks if <92% |
| **Approval gate design:** Quality gates run sequentially | Slow `/review` command | 5-10s latency on quality checks | Gates run in parallel; instant `/review` feedback |

## STRUCTURAL CHANGES

### 1. Agent Hierarchy (New)

**v3.1:** Flat agent list, sub-agent relationships unclear
```
guide-agent
├── research-agent (with sub-agents unclear)
├── scraper-agent (sub-agents overlap?)
├── creator-agent (which sub-agent applies voice?)
├── seo-agent
├── brand-agent
├── workflow-agent
└── (memory management implicit?)
```

**v3.2:** Clear 3-tier hierarchy
```
TIER 1: GATEWAY
  └─ guide-agent (router + memory manager)

TIER 2: PRIMARY EXECUTION
  ├─ research-agent (owns discovery + profiling)
  ├─ scraper-agent (owns scraping + sync state)
  ├─ creator-agent (owns content generation + comparison)
  ├─ seo-agent (owns optimization + readability)
  ├─ brand-agent (owns tone analysis + rule authoring)
  └─ workflow-agent (owns QA gates + export + archival)

TIER 3: UTILITIES
  └─ memory-manager (invoked only by guide-agent)
```

### 2. Data Ownership (New)

**v3.1:** Implicit ownership (ad-hoc writes)
```
sync-status.json ← Written by: scraper-agent? delta-sync-agent? Both?
quality-report.json ← Written by: workflow-agent? But validated by which agent?
```

**v3.2:** Explicit ownership matrix
```
sync-status.json ← OWNER: scraper-agent / sync-state-writer sub-agent
  - Read-only: guide-agent, workflow-agent
  - Write rule: Only after scrape completes + validation
  - Error rule: If write fails 3x, rollback scraped files, preserve old state

quality-report.json ← OWNER: workflow-agent / quality-checker sub-agent
  - Read-only: All agents
  - Write rule: Overwrite on each /review
  - Backup: Keep in logs/ for history

[... 8 more data files with explicit ownership ...]
```

### 3. Sub-Agent Input/Output Contracts (New)

**v3.1:** Sub-agents vague
```
delta-sync-agent:
  - Purpose: "Detect deltas"
  - Input: ???
  - Output: ???
  - Validation: ???
```

**v3.2:** Sub-agents with explicit contracts
```
delta-detector (sub-agent of scraper-agent):
  - Input: index.json (enabled competitors), sync-status.json per competitor
  - Logic: Hash URLs, compare publish dates, identify delta: {new, updated, deleted}
  - Output: delta-payload.json {competitor_slug: {new_urls: [], updated_urls: []}}
  - Validation: 0 false positives (only deltas >24h old are real), <10% false delta rate
  - ERROR: If sitemap unreachable, fallback to RSS; if both fail, mark stale
  - NEXT: Output passed to ethical-crawler sub-agent
```

### 4. Error Recovery Cascade (New)

**v3.1:** Unclear who retries
```
/sync fails
  → Log error
  → User manually retries?
```

**v3.2:** Clear ownership
```
/sync fails to detect deltas
  → scraper-agent / delta-detector owns retry
  → Retry logic: Fallback from sitemap to RSS; if both fail, check manual URL list
  → Max 3 retries with exponential backoff
  → If still fails: Log to sync-delta.jsonl as "stale", mark competitor for manual check
  → NEXT: guide-agent suggests manual review or /research competitors again
```

### 5. Quality Gate Execution (New)

**v3.1:** Sequential (slow)
```
/review
  → Check SEO score (5s)
  → Check brand tone (3s)
  → Check readability (2s)
  → Check image SEO (4s)
  → Check originality (6s)
  → Total: 20s, blocking
```

**v3.2:** Parallel (fast)
```
/review
  → [Parallel] Check SEO, brand, readability, image, originality simultaneously
  → Total: ~6s (longest-running gate), non-blocking
```
